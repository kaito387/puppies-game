### 1.2 GitHub Actions工作流详解

#### 1.2.1 完整工作流配置

文件位置: `.github/workflows/ci.yml`

#### 1.2.2 工作流执行流程

**触发条件**:

```yaml
on:
  push:
    branches: [main, develop] # 推送到这两个分支
  pull_request:
    branches: [main, develop] # 提交PR到这两个分支
```

**条件执行**:

```yaml
# Build只有在Test通过后才执行
build:
  needs: test

# Deploy只在main分支的push时执行
deploy:
  needs: build
  if: github.ref == 'refs/heads/main' && github.event_name == 'push'
```

#### 1.2.3 使用 GitHub Actions 的收益

| 收益             | 具体例子                       |
| ---------------- | ------------------------------ |
| **自动质量检查** | 不符合 lint 规则的代码自动被拒 |
| **自动测试**     | 每次 commit 都跑所有测试       |
| **自动部署**     | main 分支的 commit 自动上线    |
| **覆盖率追踪**   | Codecov 追踪覆盖率趋势         |

**配置文件**: `eslint.config.js`

## 3. 协作化体系

### 3.1 Git工作流规范

#### 3.1.1 分支模型 (Git Flow变体)

```
main (生产分支)
  ├─ 只接受来自 develop 的 merge
  ├─ 每个 commit 都是一个版本
  └─ 需要 review

develop (开发主分支)
  ├─ 团队日常开发分支
  ├─ 每个 commit 都已测试
  └─ 需要 review

feature/* (功能分支)
  ├─ 从 develop 分出
  ├─ 命名: feature/user-auth, feature/build-system
  └─ 完成后 PR 到 develop

bugfix/* (修复分支)
  ├─ 从 develop 分出
  ├─ 命名: bugfix/population-calculation
  └─ 完成后 PR 到 develop
```

#### 3.1.2 提交信息规范 (Conventional Commits)

**格式**:

```
<type>(<scope>): <subject>

<body>

<footer>
```

**类型** (type):

```
feat:     新功能 (会增加 CHANGELOG 条目)
fix:      修复 bug
docs:     文档修改
style:    代码风格修改 (不改逻辑)
refactor: 重构 (改代码但不改功能)
perf:     性能优化
test:     添加/修改测试
chore:    构建/依赖/工具修改
ci:       CI/CD 配置修改
```

#### 3.1.3 Pull Request (代码审查)

**PR模板** (`.github/pull_request_template.md`):

```markdown
## Description

这个PR做了什么？

## Related Issue

修复 #123
实现 #456

## Changes

- [ ] 新增功能
- [ ] 修复bug
- [ ] 文档更新
- [ ] 性能优化

## Testing Done

- [ ] 本地运行 `npm run test` 通过
- [ ] 本地运行 `npm run lint` 通过
- [ ] 新增了相关测试
```
